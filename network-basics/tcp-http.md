# 超文本传输协议 HTTP/0.9
* HTTP/0.9 需要很简单 用来在网络之间传递HTML超文本内容,所以也就被称为 超文本传输协议
* 基于请求响应的模式 从客户端发出请求,服务器返回数据



## HTTP/0.9 一个完整的请求流程
1. 因为HTTP都是基于TCP协议的,所以客户端先要根据IP地址 端口号  和服务器建立TCP连接,而
建立连接的过程就是TCP协议三次握手的过程
2. 建立好连接之后,会发送一个GET请求行的信息, 如 GET  /inde.html用来获取index.html
3. 服务器接收请求信息之后,读取对应的htmlwenj ，并将数据以ASII字符流返回给客户端
4. html文档传输完成后,断开连接。


* 客户端 构建请求  通过DNS查询IP地址  三次握手建立TCP连接  客户端发起请求
* 服务器处理数据阶段 服务器响应  四次挥手断开TCP连接

## HTTP/1.1
1. 改进持久连接
* http/1.0每进行一次http通信,都需要经历建立TCP连接 传输http数据和断开tcp连接三个阶段。当单个页面图片文件越来越多,下载每个文件都经历建立TCP连接  断开连接 会增加大量无谓的消耗

* `HTTP/1.1中增加了持久连接的方法,它的特点是在一个tcp链接上可以传输多个http请求,只要浏览器或者服务器没有明确断开连接,那么该tcp连接会一直保持。`
* 持久连接在 HTTP/1.1中是默认开启的,所以你不需要专门为了持久连接出http请求头设置信息,
* 如果你不想要采用持久连接,可以在http请求头中加入Connection:close。
* `目前浏览器中对于同一个域名,默认允许同时建立6个TCP持久连接`
2. 不成熟的http管线化
* 持久连接虽然能减少TCP的建立和断开次数,但是它需要等待前面的请求返回之后,才能进行下一次请求。
* 如果TCP通道中某个请求因为某些原因没有及时返回,那么就会阻塞后面的所有请求,这就是著名的`队头阻塞`的问题
* HTTP/1.1试图通过管线化的技术解决队头阻塞的问题。HTTP/1.1中管线化是指将多个http请求整批提交给服务器的技术,虽然可以整批发送请求,不过服务器依然需要根据请求顺序回复浏览器的请求。
3. 提供虚拟主机的支持
* 在http/1.0中，每个域名绑定了一个唯一的IP地址,因此一个服务器只能支持一个域名。
* 但是随着虚拟主机技术的发展,需要现在在一台物理主机上绑定多个虚拟主机，每个虚拟主机都有自己的单独的域名,这些单独域名都公用同一个IP地址。
* Host字段 用来表示当前的域名地址,这样服务器就可以根据不同的host值做不同处理。
4. 对动态生成的内容提供完美支持
* http/1.0 需要在响应头中设置完整的数据大小 Content-Length:901 这样浏览器就可以根据设置的数据大小接受数据。
* HTTP/1.1通过引入 Chunk-transfer机制来解决这个问题,服务器会将数据分割成若干个任意大小的数据块,每个数据块发送时会附上上个数据块的长度,最后使用一个零长度的块作为发送数据完成的标志。这样就提供了对动态内容的支持。
5. 客户端Cookie 安全机制

# 那么如何保证页面文件能被完整地送达浏览器呢?
##  一个数据包的'旅程'
* 互联网中的数据是通过数据包来传送的。如果发送的数据很大,那么该数据就会被拆分为很多小数据包来传输。比如你现在听的音频数据,是拆分成一个个数据包来传输的,并不是一个大的文件一次传输过来的。

1. IP:把数据包发送目的主机
* 数据包要在互联网上进行传输,就要符合 网际协议(ip)
* 计算机的地址就称为ip地址,访问任何网站实际上只是你的计算机向另外一台计算机请求信息。

* 上层将含有 '极客时间'的数据包交给网络层
* 网络层将IP头附加到数据包上,组成新的 IP数据包 并交给底层
* 底层通过物理网络将数据包传输给主机B
* 数据包被传输到主机B的网络层,在这里主机B拆开数据包的IP头信息,并将拆开来的数据部分交给上层
* 最终,含有极客时间 信息的数据包就到达了主机B的上层了。

2. UDP:把数据包送达应用程序

3. TCP:把数据完整地送达应用程序
* 如果使用UDP来传输会存在两个问题
* 数据包在传输过程中容易丢失
* 大文件会被拆分成很多小的数据包来传输,这些小的数据包会经过不同的路由,并在不同的时间到达接收端,而UDP协议并不知道如何组装这些数据包,从而把这些数据包还原成完整的文件


* 一个完整的TCP连接的生命周期包括  建立连接(三次握手)  传输数据(每个数据包确认)  断开连接(四次挥手)
* 通过TCP头的信息保证了一大块的数据传输的完整性。

* 基于http不用担心数据包丢失的问题,因为丢失和重传都是在tcp层解决的。http能保证数据按照顺序接收的。

* 构建请求  查找缓存  准备IP地址和端口  建立TCP连接   发起HTTP请求  服务器处理请求 服务器响应请求 断开TCP连接



# HTTP2.0
* HTTP/1.1为网络效率做了大量的优化,最核心的有如下三种方式:
1. 增加了持久连接
2. 浏览器为每个域名最多同时维持6个TCP持久连接
3. 使用CDN的实现域名分片机制  将一个页面的资源利用多个域名下载，提高tcp并发数量

## HTTP/1.1的主要问题
* HTTP/1.1对带宽的利用率却不理想
* 带宽是指每秒最大能发送或者接受的字节数
* 每秒能发送的最大字节数称为 上行带宽
* 每秒能够接受的最大字节数称为 下行带宽。
* 三个原因导致:
1. TCP慢启动   
* 慢启动时TCP为了减少网络拥塞的一种策略,我们是没有办法改变的。
2. 同时开启多条TCP连接,那么这些连接会竞争固定的带宽
* 资源竞争时无法区分优先级(html js css  img video) 可能会影响关键资源的下载速度
3. HTTP/1.1队头阻塞的问题
* 如果 TCP 通道中的某个请求因为某些原因没有及时返回，那么就会阻塞后面的所有请求，这就是著名的队头阻塞的问题。

## HTTP/2的多路复用
* 一个域名只使用一个TCP长连接 消除队头阻塞 以及TCP 连接之间的竞争问题。
*  通过引入二进制分帧层 就实现了HTTP的多路复用技术


1. 多路复用
2. 可以设置请求的优先级
3. 服务端推送
4. 头部压缩

## HTTP/3 甩掉TCP TLS 的包袱 构架高效网络

* HTTP2.0存在一些比较严重的与TCP协议相关的缺陷,但由于TCP协议僵化 操作系统也是导致TCP协议僵化的一个原因,tcp协议都是通过操作系统内核来实现的。

# HTTPS：让数据传输更安全
* 我们使用HTTP传输的内容很容易被中间人窃取 伪造 和篡改,通常我们把这种攻击方式称为 中间人攻击

* 总的来说,安全层有两个主要的职责:对发起HTTP请求的数据进行加密操作和对接受到http的内容进行解密操作。

* 对称加密: 所谓对称加密是指加密和解密都使用的是相同的密钥
* 非对称加密:非对称加密算法有 A B两把密钥,如果你用A密钥来加密 那么只能使用B密钥来解密;反过来 B密钥加密 只能使用A密钥来解密。


* 在https中  浏览器会将其中的一个密钥通过明文的方式发送给浏览器,我们把这个密钥称为公钥,服务器自己留下的那个密钥是 私钥
* 公钥是每个个都能获取到的,而私钥只有服务器才能知道,不对任何人公开。



* 在传输数据阶段依然使用对称加密 但是对称加密的秘钥我们采用非对称加密来传输

1. 浏览器支持的 对称加密套件类表  非对称加密套件类表 client-random
2. 服务器 保存 client-random   选择  对称加密套件  非对称加密套件。     发送浏览器   service-random   选择的对称加密套件  非对称加密套件 公钥
3. 浏览器保存 service-random 对称加密套件 非对称加密套件  公钥     随机生成 pre-master,并通过 公钥加密发送给 服务端  (由于 公钥加密的只有服务端私钥 才可以解密所以 黑客是获取不到的)
4. 服务端 client-random  service-dom  pre-master  生成  master-secret  对称加密密钥
5. 浏览器 获取 master-secret
