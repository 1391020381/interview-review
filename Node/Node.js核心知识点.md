 # 多进程解决方案:cluster模式以及PM2工具的原理介绍

 * Node.js中的cluster模式 主进程监听一个端口，子进程不监听端口，通过主进程分发请求到子进程
 
 * cluster模块应用 child_process来创建子进程，子进程通过复写掉 cluster.getServer方法，从而在 server.listen来保证只有主进程监听端口,主子进程通过IPC进行通信,其次主进程根据平台或者协议不同，应用两种不同模块（round_robin_handle.js 和 shared_handle.js）进行请求分发给子进程处理




 1. 孤儿进程 父进程先退出,子进程由pid 为1的init进程托管
 2. 僵尸进程  子进程先退出，但是父进程没有获取子进程的状态信息，导致子进程的描述符仍然保存在系统中。僵尸进程是由危害的,处理进程会以父进程的身份对僵尸进程状态进程处理。
 3. 守护进程
    - 守护进程是在 后台运行不受终端控制的进程（如输入输出）
    - 使子进程成为进程组的头
    - 中断父进程的i/o
    - 出除父进程的事件循环中对子进程的引用。



# 哪些因素会影响Node.js性能

## 代码逻辑
1. CPU密集型计算
    - 大的数据循环
    - 字符串处理转换 比如 加解密 字符串序列化等
    - 图片 视频的计算处理  比如对图片进行裁剪 缩放或者切割等。
2. 网络I/O
* 网络I/O一般不会影响主线程逻辑，往往 网络I/O请求的服务反而使瓶颈。
3. 磁盘I/O   

* 只有CPU密集计算会真正影响到Node.js服务性能
* 而网络I/O和磁盘I/O都是直接影响服务器性能，从而侧面影响到Node.js服务器性能



4. 集群服务
* master进程会存在瓶颈，因为所有的请求都必须经过master进程进行分发，同时接受处理 worker进程返回。
* 大概在2万以上的并发时,master进程会存在性能瓶颈。

# 其他
1. 内存限制
* 在32位服务器上Nodejs的内存限制时0.7G  64位服务器是1.4G
* 这个限制主要原因是Node.js垃圾回收线程在超过限制内存时,回收时长循环会大于s 从而影响性能。

* 现网我们一般会 启用多个进程,如果每个进程损耗1.4G 那么加起来可能超出服务器内存上限，从而导致服务器瘫痪。
* 其次如果内存不会超出服务器上限,而是在达到一定上限时,也就是我们上面说的0.7G 1.4G 会导致服务器重启，从而会导致接口请求失败的问题。

2. 句柄限制
* 句柄简单理解为一个 ID索引。通过这个索引可以访问到其他的资源, 文件句柄 网络I/O操作句柄。 而一般服务器句柄都是有上限的。
* 没有控制好句柄，会出现句柄泄漏问题,导致服务器异常,从而影响Node.js服务。

* 工具检测 在服务上限之前进程检测,  其次是定位具体问题。

# 优化设计: 在I/O方面应该注意哪些要点？
* I/O不会影响整体Node.js服务的性能,但是会因为I/O影响服务器系统,从而侧面影响Node.js性能。

## I/O基础介绍
* I/O在系统层面的两个阶段
1. 读取文件,将文件放入操作系统内核缓冲区
2. 将内核缓冲区拷贝到应用程序地址空间


1. 阻塞I/O
2. 非阻塞I/O
3. 多路复用I/O
* 为了解决 轮询调度的问题。 
* 我们可以将这些 I/O Socket 处理结果统一交给一个独立线程处理，当I/O Socket处理完成后,就主动告诉业务,处理完成了,不需要业务都来轮询。
* select poll epoll
* 有一个重点是这三者只会告知文件读取进入了操作系统内核缓冲区，也就是上面我们所说的第一阶段，但是第二阶段从内核拷贝到应用程序地址空间还是同步等待的
4. 信号驱动I/O
* 不需要有其他线程来处理
* 第一阶段可以由系统层面来处理，不需要独立线程来管理，但是第二阶段还是和多路复用一样。
5. 异步I/O
* 异步I/O 是连个阶段都完成啦以后，才会通知。
* 对于 Node.js 应用来说是一个异步 I/O，因此无须处理两个过程，而在 libv 内部实现，则是多线程的一个 epoll 模型
